# Assistants Module

`assistants` owns the server-side Assistant domain and persists configuration, topics, turns, and idempotent operations in `assistants.sqlite` under the OpenChamber data directory. Routes register before the generic OpenCode proxy.

`GET /api/openchamber/assistants/capability` returns `supported`, the persisted `enabled` switch, domain revision, and relay-derived stable `serverInstanceID`. `PUT /api/openchamber/assistants/settings` changes the global switch with `expectedRevision`. CRUD lives under `/api/openchamber/assistants`; topics, messages, `/new`, `/compact`, operations, and revision snapshots live under the same route family.

Every Assistant owns an immutable Inbox topic. Continuous topics reuse their OpenCode session. Stateless topics create one OpenCode session per submitted turn and preserve the logical transcript in `assistant_turn`. `GET .../messages` fetches every distinct persisted session once, follows all SDK pagination cursors, and groups assistant messages by `AssistantMessage.info.parentID === Turn.messageID`. This preserves ordinal user/assistant pairs across stateless sessions. Message sources are `composer`, `ios-share`, or `android-share`; parts use OpenCode-compatible text and bounded base64 image data URLs. Session metadata marks Assistant sessions as `openchamber.assistant`, allowing the server session index to filter them from normal sidebar summaries.

`Turn` is a discriminated DTO. User turns contain `{ id, topicID, ordinal, parentMessageID: null, phase, role: "user", kind, source, parts, assistantRevision, sessionID, messageID, operationID, createdAt, completedAt: null, error: null }`; `sessionID` and `messageID` are nullable. Assistant turns contain `{ id, topicID, ordinal, parentMessageID, phase, role: "assistant", parts, sessionID, messageID, operationID, createdAt, completedAt, error }`; `parentMessageID`, `sessionID`, `messageID`, `createdAt`, `completedAt`, and `error` are nullable. `ordinal` is the durable per-topic ordering key. `parentMessageID` links every assistant response to its persisted user message, and `messageID` carries the real OpenCode ID for each respective message. User `phase` is the linked operation state; assistant `phase` is `running`, `completed`, `failed`, or `unresolved` from its live message and transcript fetch state.

`Operation` is `{ operationID, topicID, type, state, phase, admission, result, errorCode, payloadHash, sessionID, messageID, createdAt, updatedAt }`. `topicID`, `sessionID`, `messageID`, `result`, `errorCode`, and `admission.leaseExpiresAt` are nullable at applicable lifecycle phases. Admission for `message`, `new`, and `compact` executes in one `BEGIN IMMEDIATE` transaction, with a partial unique index enforcing one running operation per topic. `phase` records recoverable external-work boundaries: `admitted`, `creating_session`, `prepared`, `submitting`, and `submitted`. Restart reconciliation uses persisted session/message IDs; only `admitted` message work is delivered. Expired uncertain work receives bounded reconciliation leases and reaches the structured `reconciliation_unresolved` terminal error, releasing its topic.

Assistant image file parts accept base64 data URLs for `image/png`, `image/jpeg`, `image/gif`, `image/webp`, `image/avif`, and native `image/heic`. The decoded limit is 8 MiB per image and 16 MiB across one submission. The common request parser allows 50 MiB JSON payloads for Assistant routes, leaving room for base64 encoding while service validation enforces decoded limits.

A server reconciliation interval scans running submitted operations at startup and while clients are offline. It resolves completion/error from matching OpenCode assistant messages. `close()` clears the interval; service construction accepts clock and interval dependencies for deterministic tests. Operations retain terminal records for seven days. `/compact` calls OpenCode `session.summarize` with the topic directory and Assistant provider/model.

Assistant workspaces and skill roots resolve only under the Assistant workspace root or configured project roots. Assistant route registration initializes project roots before serving requests and refreshes them from the authoritative settings provider on each Assistant request, revoking removed projects. Skill roots are realpath-validated catalog roots. Discovery walks only requested roots with depth, entry, time, count, and byte limits, and prompt admission stores an immutable assistant revision, prompt, provider/model, agent, mode, skill catalog, and workspace snapshot. `/new` and `/compact` execute external work only for their transaction-created admission.
