# Composer module

`document.ts` owns normalized document ranges, validation, canonical materialization, and send-plan preparation. `extensions.ts` owns each reference strategy's payload validation, decoration shape, semantics, opaque-payload budget, canonical codec, and resource identity. `reference-history.ts` owns bounded undo/redo snapshots for complete Composer reference documents. `use-composer-controller.ts` owns DraftKey-scoped local editing, store commits, history preview, submission capture/recovery, and post-commit resource deltas. Chat surfaces use the same controller and provide an explicit stable `DraftKey`; primary session and new-draft keys retain their existing durable encoding, while `surface` owners isolate secondary surfaces.

External, recovery, serialization, and materialization boundaries use full Composer validation. Browser-local edits use trusted normalization after the Draft parser or insertion validator has established typed payloads; it validates structural range and budget invariants without another opaque Paste Unicode scan. Native dictation and IME replacement ranges stay under browser ownership when reconciliation preserves their text and selection. Active IME composition suppresses imperative value, selection, and atomic-selection correction until `compositionend`, preventing iOS marked text from becoming a native blue selection. Reconciliation emits `requiresTextCorrection` from reference-range expansion, and imperative textarea correction runs only when this structural signal or caret mapping requires it. Unicode metrics use a shared bounded 400k UTF-16 LRU cache.

Add a reference kind by extending the durable draft union, then add its exhaustive strategy in `extensions.ts`. A strategy supplies its decoration and send contribution; ChatInput structurally merges the decoration into `HighlightRange` and keeps rendering, paste labels, and temporary attachment behavior locally. Chip-like mentions share one metric-safe visual contract in `inline-visual.ts`: `ComposerTriggerIconSpec` defines any trigger, icon, label, and optional suffix; `composerTriggerIconDisplay()` adds the durable em-space icon slot; `composerTriggerIconVisual()` maps the exact source display into the overlay; `composerTriggerIconText()` produces semantic delivery text. `COMPOSER_TRIGGER_ICON_SIZE_CLASS` owns the shared Composer and sent-message icon size for Session, Skill, Command, image, and attachment references. Session, Skill, and Command references use the reserved slot, so their full-size end-aligned icon and label gap stay inside textarea metrics. Legacy source text and temporary attachment citations anchor the same full-size icon to their trigger. Durable Skill references retain a slash-rule `skillName`, serialize as `[skill:name]`, and Durable Command references retain `commandName` plus a bounded `reference`, serialize as `[command:reference]`. Both use `mentionCommand` decoration, materialize from canonical tokens, and contribute no synthetic semantic; authored legacy slash skills retain their semantic delivery path.

Resource owners receive `ComposerResourceDelta` identities after successful local commits. Image strategies return `{ type: 'attachment', attachmentRefID }`; Session, Paste, Skill, and Command strategies return no resource. Store hydration and DraftKey switches establish state without a local resource delta. Atomic Composer reference deletion owns whitespace normalization and records before/after snapshots containing text, every reference kind, mentions, and selection. Browser undo/redo restores these snapshots only when its current document matches the recorded boundary, preserving Paste payloads and all Session/Skill/Command semantics together. Sent-message Session, Skill, Command, and citation references reuse the trigger/icon/label/suffix spec with a centered inline row, `mx-1` outer clearance, and bounded long-label wrapping.

Generated send chunks use `generated-reference` provenance with strategy semantics. `delivery.ts` owns authored-only chunk aggregation, attachment deduplication by normalized server path or ID, exhaustive semantic partitioning, and Session and Skill synthetic delivery parts from the owner directory snapshot. `components/chat/chatComposerDelivery.ts` is the shared chat authored compiler for direct sends and queue dispatch; it resolves agent, file, skill, session-label, and citation delivery only in authored chunks. Composer citation decoration preserves image identity with the image icon while code-selection citations retain the attachment icon. Generated references and Paste payload chunks remain byte-for-byte opaque.

Queue admission stores `content` as the queue-canonical projection and an optional `composerDocument` sidecar as the authoritative semantic body. Strict parsing requires queue-canonical sidecar serialization to equal `content`; a sidecar's visible `text` may retain compact Paste labels or Session labels. Legacy rows use text-only delivery. `describeComposerDocumentResources()` defines sidecar attachment identities, which form a subset of queue attachments; independent attachments remain valid queue resources.

Queued Composer documents may carry `composerMentions`, a bounded authored file/agent identity sidecar. It retains confirmed file identity across queueing, recovery, edits, and durable delivery. The controller creates a stable range order before commit, while strict Draft and Queue parsers reject overlap, token drift, and reference-range collisions. Skill and image behavior remains owned by Composer reference strategies.
