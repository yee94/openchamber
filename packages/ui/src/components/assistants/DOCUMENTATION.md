# Assistant UI

Assistant names whose first grapheme is an emoji use that grapheme as their avatar across the Assistant workspace and Settings. The visible name omits the leading emoji while the stored name remains intact for editing and synchronization. Emoji avatars render at three quarters of the avatar box so compound glyphs stay within the existing layout.

The Assistant UI owns the current-runtime Assistant surface. Its product information architecture is an assistant list plus the selected assistant’s standard OpenCode conversation. Desktop uses a flat Scheduled Tasks-style workspace: a soft left assistant list beside the conversation (no nested sidebar chrome), with elevated selected rows and a light border between panes. Mobile keeps the conversation full-width and uses a shared, single-line chip in the safe-area conversation header to open the same bounded bottom half-overlay pattern as model selection. The scrollable Assistant list shows avatars, modes, and semantic selection state. Each Assistant DTO carries `mode` (`continuous` | `stateless`), the current `sessionID`, and `sessionGeneration`. Stateless Assistants admit each composer turn onto a fresh OpenCode session; the surface force-rematerializes when the returned binding changes. Prior session IDs accumulate on `historySessionIDs`; `ChatContainer` prepends those transcripts with quiet “new conversation” dividers ahead of the live binding. Workspace changes bind a new session and replace the displayed transcript. `/new` binds a new session and `/compact` summarizes the current binding.

`AssistantConversationSurface` is a thin host over the shared `ChatContainer` shell. It injects the Assistant session/directory, secondary `ChatInput` surface, embedded `SessionSurfaceContext`, and host `onRevertMessage` so message/timeline revert restores into the Assistant draft partition. Stateless Assistants set `mutateSession: false`, which removes edit/revert from message and timeline actions; continuous Assistants keep mutation. The surface also provides `openSourceSession`, so each Assistant reply can jump into its underlying OpenCode session in the Chat tab for follow-up. The transcript uses the same MessageList / StatusRow / question-permission / timeline / auto-follow path as primary chat; Assistant does not keep a forked rendering tree. Assistant-only ownership stays in `AssistantView` (list, selection, binding, admission). `AssistantView` owns the complete secondary surface identity, scoped draft resources, scoped provider/model/agent/variant catalog (agents filtered via shared `chatComposerCatalog` / `filterVisibleAgents`), binding backend, command policy, activity, and shortcuts. Global Ctrl+X leader chords and agent-cycle hotkeys target the active mounted composer surface (`activeChatInputSurface`), not the primary chat tab alone. Assistant drafts and attachments use a stable transport-scoped `surfaceDraftKey` partition per Assistant. Selection PATCHes use latest-wins serialization keyed by Assistant ID, transport identity, and runtime generation; identity activation establishes the authoritative caller identity, while stale enqueue and flush callers reject with `AssistantSelectionStaleError`. Each selection flight owns an `AbortController`; retirement, disposal, and identity activation abort its in-flight conflict refresh. The selection backend verifies that identity before conflict refreshes and retries, then verifies it again after every asynchronous refresh boundary. Identity changes reject active selection and flush waiters as stale and isolate the new selection flight. A provider or model change clears the variant. Selection PATCH busy must not disable the composer textarea — Tab agent cycling stays focus-stable like primary chat. ChatInput flushes the selection before direct or queued delivery. Assistant commands retain standard custom and skill routing; fork, thread, and branch commands are unavailable. OpenCode owns every message snapshot and live event update.

Supported and enabled instances with an empty catalog show onboarding. The primary action opens Settings → Assistants, records a create intent, and focuses the assistant name field after the settings page mounts. Successful creation invalidates the Assistant snapshot; the still-mounted Assistant surface selects the new first assistant after Settings closes. Unsupported, disabled, transport-error, stale-data, assistant-disabled, and invalid-model states remain distinct.

On native iOS and Android only, the first supported, enabled Assistant-page visit shows `AssistantShareWelcome`. It is dismissed per device with `openchamber:assistant-share-welcome:v1`; the dialog explains the iOS Share Sheet, Android Direct Share, and selecting an Assistant target that receives shared content in its Inbox Topic. Settings can reopen the same dialog from the Assistants sidebar “Learn more” / “查看更多” control on any surface. On iOS, an admitted Assistant composer send donates an `INSendMessageIntent` through the native share bridge with identity and generated avatar metadata only; message content stays out of the donation. Share Extension submissions perform the same donation inside the extension, and suggested-recipient launches route by the donated `conversationIdentifier`.

`queries/assistantQueries.ts` is the current-instance HTTP and TanStack Query boundary. Snapshot and capability keys begin with the active runtime transport identity. `fetchAssistantSnapshot(signal)` performs an abortable direct snapshot read for stale selection-conflict recovery; it parses the route response without Query cache writes or retries. Ensure and new return `SessionBinding`; compact returns `{ binding, summarized: true }`; messages return `{ binding, messageID, admitted: true }`. A share operation carries top-level `sessionID` and `messageID` plus its persisted state, phase, attempt, lease, and error fields. It polls `share-operations/:operationID` with its captured runtime identity and generation; only `completed` settles delivery, while `failed` and `unresolved` retain structured errors. A consumer that needs a completed share binding reads a fresh Assistant snapshot or calls ensure. `AssistantView` calls ensure only while the active Assistant has no authoritative snapshot binding; an existing binding proceeds directly to session materialization. Assistant queries hold configuration snapshots only; standard OpenCode message snapshots stay in the directory sync store. Assistant activity and streaming presentation derive from the scoped standard OpenCode sync state after HTTP admission. Ordinary message admission uses a soft `ensureSessionRenderable` so the transcript updates in place from live sync; force rematerialization is reserved for `/new`, compact, and explicit retry. Full-body loading only covers cold empty transcripts, never a background refresh of an already-rendered conversation.

`stores/useAssistantUIStore.ts` owns current-runtime assistant selection, the transient Settings create-request revision, the client-global default share identity (`serverInstanceID + assistantID`), and persisted complete Assistant catalog partitions keyed by connection key. Persisted selection stores only the assistant ID. `MobileShareBridge` validates share targets after runtime switches, reads a fresh Assistant DTO, dispatches by assistant ID, and opens that assistant. Android share-draft handoff journals its exact runtime target through `materializing`, `materialized`, and `cancelled`; a durable draft snapshot includes a stable synthetic receipt marker, cancellation retains the journal, and successful target navigation finalizes it. Its durable outbox assigns each native share operation one OpenCode-compatible `msg_` message ID before network work; retry, resume, and WebView restoration reuse that ID, while `operationID` remains the native UUID.

Settings uses the standard split-page shell, background, border, spacing, sidebar layout, list rows, and semantic selection state shared by Agents, Providers, and Projects. Its Settings-only selection stays transient. New assistants default to the managed workspace (`workspacePath: null`); registered project options come from `useProjectsStore` and save their canonical paths. A removed project referenced by an existing assistant remains visible as a read-only current-path option until the user chooses a registered or managed workspace. Workspace changes are saved through the Assistant update route, where the server creates and binds the replacement session. Assistant UI contracts and mutation drafts carry no skill-root field.

Focused tests verify desktop and mobile menu order, independent scheduled-task visibility, empty-catalog Settings routing and create-field focus, bounded mobile chip selection, split Settings structure and responsive workspace selection, removal of topic and skill-root UI, semantic Assistant theme tokens, Assistant secondary surface isolation, and shared `ChatContainer` host / `ChatInput` consumption. Keep standard chat behavior unchanged when extending these shared surfaces.
