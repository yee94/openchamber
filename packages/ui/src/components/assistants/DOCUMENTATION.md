# Assistant UI

The Assistant UI owns the current-runtime Assistant surface. `AssistantView` presents the assistant list, topic list, and logical long conversation on desktop and as three sequential mobile stages. Every assistant uses its immutable `inboxTopicID` as Inbox. Ordinary topics support create, rename, and archive. The composer admits text and image data URLs and maps the only client commands, `/new` and `/compact`, to their dedicated routes.

`queries/assistantQueries.ts` is the current-instance HTTP and TanStack Query boundary. Snapshot, topic, turn, and capability keys begin with the active runtime transport identity. Query failures retain an existing cache snapshot and remain visible as stale/offline state; a successful empty response is authoritative. Operation POST responses require an explicit operation DTO; `admitted` and `running` reconcile through the operation route until `completed` or `failed`, and transport exceptions reconcile by the stable operation ID. Operation reconciliation is single-flight per captured transport identity plus operation ID, with generation validation before and after every GET.

`stores/useAssistantUIStore.ts` owns current-runtime assistant/topic selection, the client-global default share identity (`serverInstanceID + assistantID`), and persisted complete Assistant catalog partitions keyed by connection key. Failed catalog refreshes preserve a partition; successful empty snapshots replace it. Removing a connection evaluates the remaining complete catalog before clearing a matching composite default identity. `MobileShareBridge` refreshes current partitions, publishes their aggregate to the Capacitor share plugin, durably admits native share envelopes, validates the target after each runtime switch, and dispatches idempotent Assistant message operations. Drain has a global flight, one fair delivery worker, and an operation-ID delivery flight, so only one share operation switches the global runtime at a time. Native cleanup retries each `server-completed → native-acked → files-released` phase; a retained cleanup failure preserves its delivered operation and yields the queue without another server dispatch. `readAssistantSnapshot` and `ensureAssistantSnapshot` expose current snapshot data. `openAssistant(assistantID, topicID)` is the navigation entry point after a delivered share.

The server DTO is runtime-validated: malformed capability, snapshot, assistant, topic, turn, and operation responses raise structured contract errors while TanStack Query retains a prior snapshot. Capability carries `supported`, enablement, revision, and nullable server instance ID; assistants carry revision, enablement, prompt, workspace, skill roots, provider/model, agent, mode, Inbox ID, nullable creation timestamp, timestamps, and tombstone; topics carry assistant/session identity and revision. Turns discriminate by `role`: user turns always carry `parentMessageID: null`, numeric creation time, `completedAt: null`, and `error: null`; assistant turns carry nullable parent/message/session IDs and nullable creation, completion, and error fields. Both roles carry phase, parts, operation ID, and a nullable message ID. Operations carry their semantic operation fields plus nullable topic, session, message, result, error code, and admission lease fields. The transcript polls while an assistant turn remains unresolved.
