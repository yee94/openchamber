# Chat components

`ChatInput` owns the complete composer DOM for primary chat and explicit chat surfaces. `chatInputSurface.ts` defines a discriminated primary/secondary contract with stable surface and draft identity, active state, session and directory, selection flush/capture, scoped model/agent/variant catalog readiness, typed backend operations, command policy, activity, delivery target, and scoped composer resources. Selection values carry provider, model, agent, and variant; catalogs carry variants for that selected model plus their readiness state. Secondary model controls and mobile agent cycling consume the surface catalog, keeping workspace catalogs isolated from global configuration. Every backend request carries surface ID, transport identity, runtime generation, session, directory, draft key, captured model selection, delivery parts, and system context. Server-queued Assistant items retain required compiled `deliveryParts` entries as text or file payloads, while canonical `content` remains the queue display projection. Queued dispatch uses the typed `backend.sendQueued` port with its captured queue scope, while durable UI queue scope keys retain delivery target and runtime generation. `QueuedMessageChips` receives that resolved bound scope plus draft resource callbacks, so queue selection and mutations stay within the owning surface. Attachment lists receive the surface resource snapshot and removal port as props. Secondary identity validation throws before dispatch when a required scope field is absent.

Every surface supplies a stable `DraftKey`; `surface` draft owners isolate secondary drafts and attachments from primary session and new-draft keys. Command policy filters autocomplete, guards command selection, and validates the final authored slash command before model routing, queue admission, and local command resource consumption. Hidden surfaces set `active: false`; the shell avoids mounting the composer runtime, queries, subscriptions, hydration, and resource consumers.

`ChatContainer` accepts an optional `host` (`chatContainerHost.ts`) for embedded transcripts. Hosted mode skips the primary session-view cache, injects session/directory/composer surface/`SessionSurfaceContext`, merges `onRevertMessage` into that surface so message actions and the timeline restore into the host draft partition, and disables primary-only draft/prompt-navigator/return-to-parent features unless explicitly re-enabled.

Hosted Assistant history uses the Assistant HTTP infinite query instead of per-session sync materialization. Its key and request closure capture transport identity plus runtime generation, and validate currentness before and after every page request. It preserves server `Message`/`Part` references plus source session and nullable directory while adding semantic session dividers ahead of the live binding. Each historical message with a source session receives a source-scoped read-only `SessionSurfaceContext`; known directories scope file paths, tool actions, and source-session navigation to the archived workspace, while unknown legacy directories keep these directory-dependent actions unavailable. The shared timeline controller loads current-session sync history first and then asks the host for the next Assistant history page, so desktop near-top loading and the mobile button share one completion/loading signal through small and empty cursor pages.
