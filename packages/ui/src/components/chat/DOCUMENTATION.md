# Chat components

`ChatInput` owns the complete composer DOM for primary chat and explicit chat surfaces. `chatInputSurface.ts` defines a discriminated primary/secondary contract with stable surface and draft identity, active state, session and directory, selection flush/capture, scoped model/agent/variant catalog readiness, typed backend operations, command policy, activity, delivery target, and scoped composer resources. Selection values carry provider, model, agent, and variant; catalogs carry variants for that selected model plus their readiness state. Secondary model controls and mobile agent cycling consume the surface catalog, keeping workspace catalogs isolated from global configuration.

Primary selection ownership: ModelControls always receives the surface `selectionAdapter`, so session restore is coordinated by primary `ChatInput` (not ModelControls). On session/messages/catalog/config-scope change, primary resolves history (latest user message agent/provider/model/variant) then same-client selection-store memory, applies via `applyPrimaryComposerSessionRestore` (session memory only — never Project `saveAgentModelSelection` / `lastSelectedAgentName`). Restore effect and flush both require `activeDirectoryKey === getConfigDirectoryKey(sessionDirectory)` so a Project B session never writes into Project A's active config; flush activates the session directory when needed, then awaits `ensureSessionRenderable` with that directory and re-validates runtime generation/transport/session/session-directory/active-config-key/edit revision before apply. Explicit user picks use `applyPrimaryComposerSelectionChange` and bump an edit revision so in-flight flush cannot overwrite them. After flush, primary send/queue capture reads live `useConfigStore.getState()` (scoped by session Project key); secondary continues surface selection values. New queue admission requires a complete `sendConfig` (providerID+modelID) and keeps composer resources when capture fails; legacy rows missing `sendConfig` may still resolve live fallback at dispatch. Queued items with `sendConfig` keep that capture and skip flush when queued-only.

Every backend request pins surface ID, transport identity, runtime generation, session, and directory **before** selection flush so a mid-flight Session switch cannot retarget the body. Primary `backend.send` rejects stale runtime generation/transport (restoreFailedSubmission path), and injects request `sessionId`/`directoryHint` into `sendMessage` for existing sessions while draft `sessionID=null` keeps materialization. Captured provider/model/agent/variant stay first-submit authoritative. Server-queued Assistant items retain required compiled `deliveryParts`; queued dispatch uses `backend.sendQueued` with captured queue scope.

Every surface supplies a stable `DraftKey`; `surface` draft owners isolate secondary drafts and attachments from primary session and new-draft keys. Command policy filters autocomplete, guards command selection, and validates the final authored slash command before model routing, queue admission, and local command resource consumption. Hidden surfaces set `active: false`; the shell avoids mounting the composer runtime, queries, subscriptions, hydration, and resource consumers.

`ChatContainer` accepts an optional `host` (`chatContainerHost.ts`) for embedded transcripts. Hosted mode skips the primary session-view cache, injects session/directory/composer surface/`SessionSurfaceContext`, merges `onRevertMessage` into that surface so message actions and the timeline restore into the host draft partition, and disables primary-only draft/prompt-navigator/return-to-parent features unless explicitly re-enabled.

Hosted Assistant history uses the Assistant HTTP infinite query instead of per-session sync materialization. Its key and request closure capture transport identity plus runtime generation, and validate currentness before and after every page request. It preserves server `Message`/`Part` references plus source session and nullable directory while adding semantic session dividers ahead of the live binding. Each historical message with a source session receives a source-scoped read-only `SessionSurfaceContext`; known directories scope file paths, tool actions, and source-session navigation to the archived workspace, while unknown legacy directories keep these directory-dependent actions unavailable. The shared timeline controller loads current-session sync history first and then asks the host for the next Assistant history page, so desktop near-top loading and the mobile button share one completion/loading signal through small and empty cursor pages.
